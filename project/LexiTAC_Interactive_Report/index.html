INTRODUCTION

Bridging Compiler Theory
with Interactive Practice

Welcome to the interactive report for LexiTAC. This section outlines the background and motivation behind our compiler front-end project. We built a modular compiler front-end from scratch using Flex, Bison, and C to demystify how raw source text is transformed into Three-Address Code (TAC).

🎯

The Problem

Students in Compiler Design courses often struggle with understanding tokenization, symbol tables, and parse tree evaluations. Existing CLI tools are too opaque, while production compilers are too complex.

💡

Our Solution

LexiTAC implements a fully functional front-end that processes arithmetic assignment expressions, enforcing operator precedence and generating intermediate code, paired with a web-based visualization.

🛠️

Core Technologies

Built utilizing industry-standard tools: Flex (Lexical Analysis), Bison (LALR(1) Syntax Analysis), and C11 standard, fully automated via GNU Make.

PERFORMANCE ANALYSIS

Complexity & Efficiency

This section illustrates the operational efficiency of the LexiTAC system. All phases (Lexical Analysis, Parsing, and TAC Generation) operate in linear time complexity O(n) relative to the input length, ensuring high efficiency.

Time Complexity

O(n)

Lexer, Parser, and TAC emission all scale linearly with input expression length.

Space Complexity

O(k) & O(t)

Symbol table maxes at 256 variables (k). Parse stack and temporaries (t) scale proportionally to nesting depth.

EP1: Depth of Knowledge

DFA/NFA & LALR

Requires understanding of formal language theory, DFA construction, parsing algorithms, and intermediate code generation.

EP2: Conflicting Requirements

CLI vs Web

Balances correctness (no shift-reduce conflicts) with simplicity (student-understandable code), and CLI power with browser accessibility.

EP4: Familiarity of Issues

Memory Safety

Addressed classic compiler construction challenges: token ambiguity, parser conflicts, memory safety, and cross-language logic replication.
